# Phase 7 Learning Architecture

## 1. Purpose

This document finalizes the Phase 7 learning architecture for Agentic AI AWR Advisor. It consolidates Phase 7A through Phase 7J into one navigable architecture view for learning boundaries, outcome pattern mining, learning candidates, semantic reviewer-assist context, governance review, dashboard visibility and interactivity, CLI operations, and local validation.

Phase 7 is a learning architecture, not runtime self-modification. The deterministic runtime remains authoritative. Learning is candidate-based, human-governed, and proposal-only until approved and implemented separately through controlled future work.

## 2. Scope

Phase 7 covers governed learning candidates derived from deterministic and governed local records. It includes local outcome pattern mining, deterministic candidate generation, optional semantic candidate context for human reviewers, local governance transitions, read-only dashboard visibility, read-only dashboard exploration, local CLI wrappers, and a consolidated local validation harness.

Phase 7 does not change parser behavior, parser output, loader behavior, scoring logic, scoring weights, trend or anomaly logic, decision logic, recommendation logic, recommendation ranking, Phase 4I output contracts, runtime activation, database schema, generated dashboard artifacts, or production readiness certification.

## 3. Phase 7 Architecture Summary

Phase 7 adds a governed learning loop around existing deterministic runtime truth. The loop observes local governed evidence, mines repeatable patterns, converts patterns into proposal-only learning candidates, enriches candidates with optional non-authoritative semantic context, and supports human review without activating runtime changes.

The learning system is intentionally separated from runtime truth paths. Parser, scoring, decision, recommendation, and `scripts/run_analysis.py` remain protected. Phase 7 learning modules are not imported by runtime truth paths, and candidates retain `runtime_influence=false` and `requires_human_review=true`.

## 4. Layered Architecture

### Deterministic Runtime Truth Layer

The deterministic runtime truth layer is Phase 4 and the validated Phase 4I contract. It includes parsing, feature engineering, scoring, trend/anomaly analysis, decision logic, recommendation logic, and output contract generation. The deterministic runtime remains authoritative for diagnostic truth, historical truth, and recommendation truth.

Phase 7 consumes runtime and memory outputs as local evidence only. It does not feed candidates back into parser, scoring, decision, recommendation, or dashboard truth.

### Governed Memory Layer

The governed memory layer is Phase 6. It persists structured run memory, actions, outcomes, feedback, unknown signal review state, knowledge requests, and approved knowledge artifacts under explicit governance. Phase 7B outcome pattern mining can observe local memory-shaped JSON records, but pattern records are observational only and are not learning candidates by themselves.

Governed memory remains auditable and human-controlled. Phase 7K does not add DB writes or persistence changes.

### Semantic Reviewer-Assist Context Layer

The semantic reviewer-assist context layer provides optional local context for candidate reviewers. Semantic context is reviewer-assist only. Semantic context is non-authoritative and cannot change candidate confidence, candidate status, source evidence, structured sources, runtime influence, or deterministic runtime output.

Phase 7E uses local semantic records only. It does not require Oracle Agent Memory, a semantic recall service, OCI, ADB, network access, or LLM calls.

### Learning Candidate Layer

The learning candidate layer contains deterministic candidate records generated from outcome patterns. Candidates are proposal-only until approved and implemented separately. Candidate records retain `runtime_influence=false`, `requires_human_review=true`, explicit source evidence, structured sources, affected component metadata, rationale, confidence, lifecycle status, and optional semantic context.

Candidate generation does not approve, implement, materialize, persist, or activate runtime behavior.

### Dashboard Exploration Layer

The dashboard exploration layer displays learning and governance context in read-only form. Dashboard interactivity is read-only and browser-side. It supports exploratory selection, cross-screen propagation, and local visibility without changing diagnostic truth, historical truth, recommendation truth, governance state, candidate status, backend state, or generated HTML unless explicitly regenerated by normal dashboard generation.

Learning candidates remain proposal/review context only and are kept out of diagnostic and recommendation evidence.

### CLI Operations Layer

The CLI operations layer exposes local deterministic learning commands under `scripts/awr_memory_cli.py learning ...`. CLI learning commands are local and actor-gated where review transitions are requested. They operate on local JSON inputs and produce local JSON or human-readable output.

The CLI does not write to Oracle DB, call OCI, call Oracle Agent Memory, call a semantic recall service, call an LLM, make network calls, or activate candidates.

## 5. Phase 7A Learning Boundary

Phase 7A defines the learning boundary and candidate lifecycle. It establishes that learning is candidate-based, human-governed, non-runtime-mutating, and subordinate to deterministic runtime truth.

Phase 7A confirms no autonomous parser changes, no autonomous scoring changes, no autonomous recommendation changes, no runtime self-modification, and no runtime activation.

## 6. Phase 7B Outcome Pattern Mining

Phase 7B adds deterministic local outcome pattern mining over memory-shaped JSON records. It identifies repeatable observational patterns such as repeated unknown signals or repeated recommendation/outcome combinations.

Outcome patterns are not learning candidates. They are local, deterministic, observational, and carry `runtime_influence=false`.

## 7. Phase 7C Learning Candidate Model

Phase 7C adds the learning candidate record model. The model captures proposal metadata, source evidence, structured sources, affected component, confidence, rationale, lifecycle status, reviewer metadata, materialization reference, and safety flags.

Candidate records are proposal-only and require human review. They do not alter runtime truth.

## 8. Phase 7D Candidate Generation Engine

Phase 7D adds deterministic conversion from outcome patterns into learning candidates. The engine maps observed pattern types to candidate types such as parser mapping candidates or recommendation refinement candidates.

The engine does not approve, implement, persist, materialize, activate, or apply candidates. It only creates reviewable proposal records.

## 9. Phase 7E Semantic Candidate Context

Phase 7E attaches optional semantic context to candidates for reviewer assistance. Semantic context is reviewer-assist only, optional, non-authoritative, not source evidence, and not a substitute for deterministic evidence.

Semantic context cannot change confidence, status, candidate type, source evidence, structured sources, or runtime influence.

## 10. Phase 7F Learning Governance Bridge

Phase 7F adds local deterministic governance transitions for candidate review. Review actions require an actor and preserve `runtime_influence=false` and `requires_human_review=true`.

Approval means approved for implementation only. It is not runtime activation, not runtime integration, and not uncontrolled autonomous learning.

## 11. Phase 7G Dashboard Learning Visibility

Phase 7G adds read-only dashboard visibility for learning candidates and learning safety state. It surfaces proposal state, review requirements, runtime influence, and governance posture without adding approval controls or write controls.

Dashboard learning visibility keeps learning candidates out of diagnostic evidence and recommendation truth.

## 12. Phase 7H Dashboard Interactivity

Phase 7H adds read-only dashboard interactivity across Screen 1 through Screen 6. It provides browser-side exploratory selection, focused evidence panels, and cross-screen selection propagation.

Dashboard interactivity is read-only. It adds no backend writes, no API calls, no approval controls, no write controls, no truth mutation, and no candidate status mutation.

## 13. Phase 7I Learning CLI Commands

Phase 7I adds local learning commands to `scripts/awr_memory_cli.py`. The commands cover status, patterns, candidates, candidate detail, semantic context, review, export, and validation.

CLI learning commands are local and actor-gated. Review commands require an actor, approval remains approved for implementation only, and all candidate/governance outputs preserve no runtime activation.

## 14. Phase 7J Validation Harness

Phase 7J adds `scripts/run_phase7_validation.py`, JSON output, validation matrix documentation, and validation harness documentation. The harness validates Phase 7A through Phase 7I locally and deterministically.

The validation harness confirms local execution, no DB dependency, no network dependency, no Oracle Agent Memory dependency, no runtime activation, deterministic runtime authority, semantic reviewer-assist boundaries, read-only dashboard interactivity, local CLI behavior, and import isolation.

## 15. Runtime Truth Boundary

The deterministic runtime remains authoritative. Phase 7 cannot change parser output, scoring output, trend/anomaly output, decision output, recommendation output, historical truth, diagnostic truth, or Phase 4I output contract.

No Phase 7 candidate has runtime influence. No runtime activation exists.

## 16. Semantic Context Boundary

Semantic context is reviewer-assist only. Semantic context is non-authoritative. Semantic context is not source evidence, does not affect candidate scoring, does not affect deterministic scoring, and does not affect recommendation truth.

Phase 7 uses local semantic records only and has no live semantic service dependency.

## 17. Governance Boundary

Governance remains human-governed. Review transitions are local candidate lifecycle transitions and require explicit actor attribution where review state changes occur.

Approval is approval for implementation only. Implementation, validation, and closure remain explicit status changes, not runtime activation.

## 18. Dashboard Boundary

The dashboard remains a presentation and exploration layer. It can display deterministic truth, governed memory context, semantic reviewer-assist context, and learning candidates, but it cannot alter backend truth.

Dashboard interactivity is read-only and cannot change parser output, diagnostic truth, historical truth, recommendation truth, governance state, candidate status, or runtime behavior.

## 19. CLI Boundary

CLI learning commands operate locally on JSON input and local module outputs. They do not create backend writes, DB writes, API calls, OCI calls, Oracle Agent Memory calls, semantic recall service calls, LLM calls, or runtime activation.

Review commands are actor-gated. Read commands are local and deterministic.

## 20. Non-Goals

Phase 7 does not implement autonomous learning, uncontrolled autonomous learning, automatic parser updates, automatic scoring updates, automatic recommendation updates, runtime activation, production readiness certification, Phase 7L, Phase 8 adaptive runtime behavior, DB write behavior, OCI dependencies, ADB dependencies, live Oracle Agent Memory dependencies, semantic recall service live dependencies, LLM calls, API calls, dashboard write controls, CLI behavior changes beyond Phase 7I, parser behavior changes, scoring changes, decision changes, or recommendation changes.

## 21. Acceptance Summary

Phase 7 architecture is accepted when deterministic runtime remains authoritative; learning remains candidate-based and human-governed; candidates remain proposal-only until approved and implemented separately; semantic context is reviewer-assist only and non-authoritative; dashboard interactivity is read-only; CLI learning commands are local and actor-gated; no runtime activation exists; and no parser/scoring/decision/recommendation behavior changes are introduced.
